# Контракты API (API Contracts)

Платформа предоставляет REST API для взаимодействия фронтенда и внешних систем. Все эндпоинты имеют префикс `/api/v1/`.

## 🔐 Аутентификация (`/auth/`)

| Метод | Эндпоинт          | Описание                                              |
| :---- | :---------------- | :---------------------------------------------------- |
| POST  | `/auth/register/` | Регистрация (B2C — сразу, B2B — требует верификации). |
| POST  | `/auth/login/`    | Получение Access/Refresh токенов.                     |
| POST  | `/auth/refresh/`  | Обновление Access токена (с ротацией Refresh).        |
| POST  | `/auth/logout/`   | Инвалидация токена (Blacklist).                       |

## 🛍️ Каталог товаров (`/products/`)

| Метод | Эндпоинт                   | Описание                                                                 |
| :---- | :------------------------- | :----------------------------------------------------------------------- |
| GET   | `/products/`               | Список товаров с фильтрацией (категория, бренд, цена, флаги).            |
| GET   | `/products/{slug}/`        | Детальная информация о товаре, включая варианты (SKU).                   |
| GET   | `/products/search/`        | Полнотекстовый поиск (PostgreSQL FTS).                                   |
| GET   | `/products/visible-brands/` | ID брендов, у которых есть товары по текущим фильтрам каталога; параметр `brand` игнорируется. |
| GET   | `/categories/`             | Список категорий и исходная иерархия для общих сценариев.                |
| GET   | `/categories-tree/`        | Публичное дерево витрины: прямые дети `СПОРТ` как корневые узлы, без служебных категорий и категорий-заглушек. |
| GET   | `/brands/`                 | Список активных брендов; `has_stock=true` возвращает только бренды с активными товарами и вариантами `stock_quantity > 0`. |
| GET   | `/brands/featured/`        | Избранные бренды для главной страницы; `has_stock` не применяется.       |

### Поведение фильтра брендов в каталоге

- `/catalog` первично загружает бренды через `/brands/?has_stock=true`, поэтому бренды без товаров в наличии не появляются в фильтре.
- При активном режиме `in_stock=true` сайдбар дополнительно сужается через `/products/visible-brands/` по текущим фильтрам (`category_id`, цена, поиск и т.д.).
- При `in_stock=false` динамическое сужение брендов не выполняется: отображается полный список из первичного `has_stock=true`-гейта.

## 🛒 Корзина и Заказы (`/cart/`, `/orders/`)

| Метод | Эндпоинт         | Описание                                                                 |
| :---- | :--------------- | :----------------------------------------------------------------------- |
| GET   | `/cart/`         | Содержимое текущей корзины (по JWT или сессии).                          |
| POST  | `/cart/add/`     | Добавление `variant_id` в корзину.                                       |
| POST  | `/orders/`       | Создание заказа из корзины для авторизованного пользователя; `customer_code` присваивается автоматически, если отсутствует. |
| GET   | `/orders/`       | Список только клиентских master-заказов пользователя.                    |
| GET   | `/orders/{id}/`  | Детали master-заказа с агрегированными позициями sub-orders.             |
| POST  | `/orders/{id}/cancel/` | Отмена master-заказа; субзаказы отменяются каскадно.               |

- `order_number` в API хранится и передаётся в каноническом виде: `CCCCCYYNNN` для master-заказа и `CCCCCYYNNNS` для sub-order.
- UI-форматирование номера (`CCCC-YYNNN` / `CCCCC-YYNNN-S`) выполняется на уровне представления и не используется как API payload.

## 📰 Подписка и согласия (`/subscribe/`, `/unsubscribe/`, `/pages/`)

| Метод | Эндпоинт | Описание |
| :---- | :------- | :------- |
| POST | `/subscribe/` | Подписка на email-рассылку с обязательным согласием ПДн. |
| POST | `/unsubscribe/` | Отписка от рассылки. |
| GET | `/pages/{slug}/` | Публичная страница (например, `privacy-policy`). |

### Контракты subscribe / unsubscribe

**`POST /api/v1/subscribe/`**

Payload:
```json
{ "email": "user@example.com", "pdp_consent": true }
```
- `pdp_consent` — обязательное поле, принимается только `true` (JSON boolean). Без него — `400 Bad Request`.
- Всегда возвращает **`200 OK`** (новая подписка или already_subscribed неразличимы — защита от email enumeration).
- Тело успешного ответа: `{"message": "Вы успешно подписались на рассылку", "email": "<email>"}`.
- `400` — невалидный email или отсутствующий/ложный `pdp_consent`.
- `429 Too Many Requests` — throttle `SubscribeRateThrottle` (30/min, scope `"subscribe"`).
- `503` — сбой БД при записи согласий.

**`POST /api/v1/unsubscribe/`**

Payload: `{ "email": "user@example.com" }`
- Всегда возвращает **`200 OK`** (успешная отписка и «email не найден» неразличимы — защита от email enumeration).
- Тело: `{"message": "Запрос на отписку обработан", "email": "<email>"}`.
- `400` — невалидный email.
- `429 Too Many Requests` — throttle `UnsubscribeRateThrottle` (30/min, scope `"unsubscribe"`).
- `503` — сбой БД.

**`GET /api/v1/pages/privacy-policy/`**

- Возвращает опубликованную страницу Политики ПДн (управляется через Django Admin, ISR кэш 1ч).
- `404` если страница не найдена или `is_published=False`.
- Frontend-маршрут: `/privacy-policy` (Server Component, ISR revalidate 3600).

## ⚙️ Особенности реализации (Frontend)

- **api-client.ts**: Базовый клиент на Axios.
- **SSR Handling**: Фронтенд определяет окружение (Server/Client) и использует соответствующий `baseURL`.
- **Error Handling**: Реализован механизм автоматического повтора запросов (`retry`) при 500/503 ошибках.
