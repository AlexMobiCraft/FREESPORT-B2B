# Анализ `orders.xml` из CommerceML 1C

> Основано на локальном архиве `backend/var/1c_exchange/logs/*_orders.xml` и на текущем импортере
> `backend/apps/orders/services/order_status_import.py`.

## Краткий вывод

- В выборке есть две версии схемы: `ВерсияСхемы="3.1"` и `ВерсияСхемы="2.09"`.
- `2.10` в архиве не встретилась.
- Основной контейнер для документов заказов - `<Контейнер>`.
- Для импорта статусных изменений надежнее `Номер`, чем `Ид`: `Номер` есть во всех документах выборки,
  `Ид` отсутствует в 1080 из 1436 документов.
- Наблюдаемый inbound-реквизит статуса - `СтатусЗаказа`.
- Наблюдаемый outbound-реквизит статуса - `Статус заказа`; он относится к экспортным ответам, но парсер
  нормализует имена реквизитов и читает оба варианта (см. раздел «Как это использует backend»).
- Реквизиты `ДатаОплаты` / `ДатаОтгрузки` в этой выборке не встретились, но текущий парсер их поддерживает
  как опциональные поля с вариантами написания через пробел.

## Статистика выборки

Из локального архива `backend/var/1c_exchange/logs` было просмотрено 1359 файлов `*_orders.xml`:

- 1152 файла содержат хотя бы один `<Документ>`.
- 1436 документов были разобраны как валидные XML-документы.
- 87 файлов содержат служебный маркер `<not valid xml!!!`.
- 120 файлов - валидные XML без `<Документ>`.

**Примечание:** таблицы ниже построены на основе 1272 файлов, для которых удалось определить версию схемы и наличие контейнера. 87 невалидных XML исключены из подсчёта.

| --- | ---: |
| `3.1` | 1222 |

По контейнеру (из 1272 валидных файлов):

## Что реально встречается в документах

### Идентификаторы

- `<Номер>` присутствует во всех документах выборки.
- `<Ид>` присутствует не во всех документах: 1080 документов из 1436 пришли без него.
- Для импорта это означает, что `<Номер>` должен считаться основным ключом, а `<Ид>` - fallback.

Текущий `OrderStatusImportService` именно так и работает:

1. ищет заказ по `<Номер>`;
2. если не нашёл, пытается разобрать `<Ид>` формата `order-{id}`;
3. если оба поля присутствуют, дополнительно проверяет, что они не конфликтуют.

### Статусные реквизиты

В архиве встретились два разных имени статусного реквизита:

- `СтатусЗаказа` - inbound-формат, который читает импорт статусов;
- `Статус заказа` - outbound-формат для экспортных ответов сайта.

Наблюдаемые значения `СтатусЗаказа` (из 1436 разобранных документов):

| Значение 1С | Кол-во в выборке | Статус FREESPORT |
| --- | ---: | --- |
| `Подтвержден` | 681 | `confirmed` |
| `Отгружен` | 494 | `shipped` |
| `Отменен` | 87 | `cancelled` |
| `Доставлен` | 82 | `delivered` |
| **Прочие/отсутствует** | **92** | **различные или пусто** |

Из 1436 документов 1344 имеют признанное значение `СтатусЗаказа` из таблицы выше; остальные 92 либо содержат outbound-реквизит `Статус заказа` (80+4=84), либо не имеют статусного реквизита вовсе (8 документов).

Дополнительно текущий код импорта поддерживает:

| Значение 1С | Статус FREESPORT | Комментарий |
| --- | --- | --- |
| `ОжидаетОбработки` | `processing` | Поддерживается `STATUS_MAPPING`, в реальной выборке не встретилось |
| `Возвращен` | `refunded` | Поддерживается `STATUS_MAPPING`, в реальной выборке не встретилось |

Наблюдаемые значения `Статус заказа` (outbound-реквизит, 84 документа):

| Значение 1С | Кол-во в выборке | Комментарий |
| --- | ---: | --- |
| `Не согласован` | 80 | Outbound-статус, попадает в парсер и отклоняется как `SKIPPED_UNKNOWN_STATUS` (не в `STATUS_MAPPING`) |
| `Новый` | 4 | Outbound-статус, попадает в парсер и отклоняется как `SKIPPED_UNKNOWN_STATUS` |

## Путь к данным

Ниже - практические XPath-подобные пути, которые соответствуют текущей структуре:

- Идентификатор заказа: `/КоммерческаяИнформация/Контейнер/Документ/Ид`
- Номер заказа: `/КоммерческаяИнформация/Контейнер/Документ/Номер`
- Дата документа: `/КоммерческаяИнформация/Контейнер/Документ/Дата`
- Статус inbound: `/КоммерческаяИнформация/Контейнер/Документ/ЗначенияРеквизитов/ЗначениеРеквизита[Наименование='СтатусЗаказа']/Значение`
- Статус outbound: `/КоммерческаяИнформация/Контейнер/Документ/ЗначенияРеквизитов/ЗначениеРеквизита[Наименование='Статус заказа']/Значение`
- Дата оплаты: `/.../ЗначениеРеквизита[Наименование='ДатаОплаты' or Наименование='Дата Оплаты']/Значение`
- Дата отгрузки: `/.../ЗначениеРеквизита[Наименование='ДатаОтгрузки' or Наименование='Дата Отгрузки']/Значение`

## Как это использует backend

`OrderStatusImportService` нормализует имена реквизитов через удаление пробелов и приведение к нижнему регистру (`re.sub(r"\s+", "").lower()`). Это означает:

- **Для статуса:** `СтатусЗаказа`, `Статус Заказа`, `Статус заказа` и другие варианты написания всё равно читаются как один ключ `статусзаказа`.
  - inbound-документы обычно используют `СтатусЗаказа` (1344 документа).
  - outbound-документы используют `Статус заказа` (84 документа) — они также попадают в парсер и отклоняются как `SKIPPED_UNKNOWN_STATUS`, так как значения `Не согласован`/`Новый` не входят в `STATUS_MAPPING`.
- **Для дат:** парсер принимает `ДатаОплаты`, `Дата Оплаты`, `ДатаОтгрузки` и `Дата Отгрузки` — все варианты нормализуются в один ключ.
- Даты оплаты/отгрузки в архиве не обнаружены, поэтому на практике они остаются опциональным расширением контракта, а не обязательным полем.
