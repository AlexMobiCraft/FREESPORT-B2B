# Контракты Backend API

**(Обновлённый срез: 2026-05-07)**

## Базовый префикс

`/api/v1/`

## Каталог (`/products`, `/categories`, `/brands`)

- `GET /products/`: Список товаров с фильтрацией и пагинацией.
- `GET /products/{slug}/`: Детальная карточка товара.
- `GET /products/visible-brands/`: Возвращает `{"brand_ids": [...]}` для брендов, у которых есть товары по текущим фильтрам каталога. Параметр `brand` намеренно игнорируется, чтобы фильтр брендов не сужал сам себя.
- `GET /categories/`: Список категорий и исходная иерархия для административных/общих сценариев.
- `GET /categories-tree/`: Публичное дерево каталога для витрины. Эндпоинт возвращает прямых детей якорной категории `СПОРТ` как корни витрины, без пагинации, и рекурсивно исключает технические категории (`uncategorized`, `onec-unresolved-category`, `Без категории`) и узлы-заглушки вида `Категория <id>`.
- `GET /brands/`: Список брендов. Параметр `has_stock=true` включает гейт по наличию: возвращаются только бренды с активным товаром и хотя бы одним вариантом `stock_quantity > 0`. `has_stock=false`, `0` или отсутствие параметра сохраняют прежний список всех активных брендов.
- `GET /brands/featured/`: Избранные бренды для главной страницы; query-параметр `has_stock` не применяется, ответ остаётся совместимым с прежним кэшируемым контрактом.
- `GET /catalog/filters/`: Динамические фильтры каталога.

### Инварианты контракта категорий

- `Category` хранит полное дерево из 1С, включая корневой якорь `СПОРТ`.
- Витрина не должна строиться от всех `parent=None`: публичные корни берутся из `/categories-tree/`.
- `СПОРТ` является внутренним якорем хранения и не отображается как отдельный публичный корень.
- Если в БД временно есть несколько активных корневых якорей `СПОРТ`, `/categories-tree/` возвращает объединение их прямых детей и пишет warning в лог; данные нужно выровнять командой восстановления.
- Архитектурное решение зафиксировано в [ADR-010: Публичная проекция дерева категорий 1С](../decisions/ADR-010-category-tree-public-projection.md).

### Инварианты контракта брендов в каталоге

- Бренд считается доступным для фильтра каталога, если существует `Product.is_active=True` с вариантом `ProductVariant.stock_quantity > 0`.
- `/brands/?has_stock=true` используется `(blue)/catalog` для первичной загрузки фильтра «Бренд»; это не меняет модель `Brand` и не требует миграций.
- `/products/visible-brands/` применяет общий `ProductFilter` к товарам и возвращает только реальные ID брендов; `brand_id IS NULL` исключается.
- При снятом чекбоксе «В наличии» frontend не вызывает `/products/visible-brands/` и показывает все бренды из первичного `/brands/?has_stock=true`.

## Корзина (`/cart`)

- `GET /cart/`: Текущая корзина пользователя или гостевой сессии.
- `DELETE /cart/clear/`: Очистка корзины.
- `POST /cart/items/`: Добавление позиции в корзину.
- `PATCH /cart/items/{id}/`: Изменение количества.
- `DELETE /cart/items/{id}/`: Удаление позиции.

## Заказы (`/orders`)

- `POST /orders/`: Создание заказа из корзины.
  - Требует авторизацию.
  - Требует валидный `customer_code` у пользователя.
  - Возвращает `OrderDetail` мастер-заказа.
- `GET /orders/`: Список только master-заказов текущего пользователя.
- `GET /orders/{id}/`: Детали master-заказа с агрегированными позициями sub-orders.
- `POST /orders/{id}/cancel/`: Отмена master-заказа с каскадной отменой sub-orders.

### Инварианты контракта заказов

- `order_number` в API всегда канонический:
  - master: `CCCCCYYNNN`
  - sub-order: `CCCCCYYNNNS`
- UI-формат (`CCCC-YYNNN` / `CCCCC-YYNNN-S`) не используется как API payload.
- Клиентские endpoint скрывают технические `sub-orders`; они используются для интеграции с 1С.

## Пользователи (`/users`, `/auth`)

- `POST /users/register/`: Регистрация пользователя.
- `POST /users/token/`: Получение JWT.
- `GET /users/me/`: Текущий профиль.
- `GET /users/company/`: B2B-профиль компании.
