# 1C Order Exchange Module PRD

## 1. Intro Project Analysis and Context

### Existing Project Overview

**Current State:**  
Проект FREESPORT реализовал транспортный уровень интеграции с 1С (модуль `onec_exchange`):

- ✅ Аутентификация (`mode=checkauth`)
- ✅ Инициализация (`mode=init`)
- ✅ Приём файлов (`mode=file`)
- ✅ Импорт товаров, цен, остатков (`mode=import`)
- ✅ Экспорт заказов через `mode=query`
- ✅ Подтверждение выгрузки заказов через `mode=success`
- ⚠️ Требуется поддержка `mode=info` для загрузки справочника статусов заказов в 1С-Битрикс

Двусторонний обмен заказами включает:

- Экспорт заказов покупателей в 1С
- Получение обновлённых статусов из 1С
- Загрузку справочника статусов сайта в настройки 1С-Битрикс через `mode=info`

### Business Goal

Обеспечить полный цикл обработки заказов:

1. Покупатель создаёт заказ на сайте
2. Заказ автоматически выгружается в 1С
3. Менеджер обрабатывает заказ в 1С
4. Обновлённый статус возвращается на сайт

## 2. Requirements

### Functional Requirements (FR)

#### FR1: Экспорт заказов (Сайт → 1С)

| ID     | Требование                                          | Комментарий                                                                              |
| ------ | --------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| FR1.1  | Реализовать `mode=query` для выгрузки заказов       | 1С запрашивает этот url, в ответ получает XML                                            |
| FR1.2  | Все новые заказы передаются **одним файлом**        | Стандарт обмена подразумевает пакетную передачу                                          |
| FR1.3  | Поддержка ZIP-сжатия (`zip=yes`)                    | Если 1С запрашивает zip, отдавать архив                                                  |
| FR1.4  | Корневой тег XML: `<КоммерческаяИнформация>`        | Стандарт CommerceML 2.10                                                                 |
| FR1.5  | Каждый заказ — тег `<Документ>`                     | Тип хоз. операции: `<ХозОперация>Заказ товара</ХозОперация>`                             |
| FR1.6  | Блок `<Контрагенты>` с **ИНН**                      | Поиск в 1С идет по ИНН или Наименованию (нужен ИНН если есть)                            |
| FR1.7  | Блок `<Товары>` с `onec_id`                         | Связь товаров осуществляется по ID из 1С                                                 |
| FR1.8  | Подтверждение (`mode=success`)                      | Помечать заказы как `sent_to_1c=true` только после этого сигнала                         |
| FR1.9  | Экспортировать в 1С только технические `sub-orders` | `master-order` остается клиентским агрегатором и напрямую в CommerceML не выгружается    |
| FR1.10 | Делить `sub-orders` по `(vat_rate, warehouse_name)` | Разные ставки НДС или разные склады внутри одной ставки должны стать разными документами |

- `<Ид>` остаётся immutable technical identifier в формате `order-{id}` и используется как fallback при импорте статусов.
- Клиентский master-order напрямую в CommerceML не выгружается.

### Актуальная логика VAT/warehouse split

При создании заказа из корзины создается один клиентский `master-order` и набор технических `sub-orders`. `sub-order` группируется по паре:

```python
group_key = (resolved_vat_rate, variant.warehouse_name)
```

Приоритет определения НДС при создании заказа:

```text
ProductVariant.vat_rate
Product.vat_rate
WAREHOUSE_RULES[warehouse_name].vat_rate
DEFAULT_VAT_RATE
```

`ProductVariant.vat_rate` остается основным каталожным источником для order/export path. `Product.vat_rate` нужен для случаев, когда ставка пришла в `goods.xml`, а варианты создаются или обновляются из `offers.xml`. После создания заказа `OrderItem.vat_rate` является snapshot-ом ставки на момент заказа.

В экспорте `vat_group` субзаказа является авторитетным источником ставки документа. Склад документа берется из товаров субзаказа только если он однозначный и соответствует `vat_group`; иначе используется fallback по `ORGANIZATION_BY_VAT` или `DEFAULT_*`.

Подробная операционная документация, правила складов и troubleshooting: [VAT-split и складской routing заказов для 1С](../integrations/1c/order-vat-warehouse-routing.md).

### Справочник статусов (`mode=info`)

Режим `mode=info` используется 1С-Битрикс не для передачи заказов, а для загрузки справочников сайта в форму настройки обмена. При нажатии кнопки **"Загрузить с сайта"** в таблице сопоставления **"Статусы заказов"** 1С ожидает XML-ответ в формате стандартного компонента Bitrix `sale.export.1c`.

```xml
<?xml version="1.0" encoding="windows-1251"?>
<Справочник>
  <Cтатусы>
    <Элемент>
      <Ид>ОжидаетОбработки</Ид>
      <Название>ОжидаетОбработки</Название>
    </Элемент>
    <Элемент>
      <Ид>Подтвержден</Ид>
      <Название>Подтвержден</Название>
    </Элемент>
    <Элемент>
      <Ид>Отгружен</Ид>
      <Название>Отгружен</Название>
    </Элемент>
    <Элемент>
      <Ид>Доставлен</Ид>
      <Название>Доставлен</Название>
    </Элемент>
    <Элемент>
      <Ид>Отменен</Ид>
      <Название>Отменен</Название>
    </Элемент>
    <Элемент>
      <Ид>Возвращен</Ид>
      <Название>Возвращен</Название>
    </Элемент>
  </Cтатусы>
  <ПлатежныеСистемы />
</Справочник>
```

Источник статусов для справочника — ключи `STATUS_MAPPING` из `backend/apps/orders/constants.py`. Значение `Не согласован` нужно включить дополнительно, если оно остаётся дефолтным статусом, который `OrderExportService` отправляет в реквизите `Статус заказа` при экспорте заказа в 1С.

## 5. Work Breakdown (Stories)

### Epic: 1C Order Exchange

#### Story 1: Order Model 1C Fields

- **Goal:** Добавить поля для отслеживания синхронизации с 1С.
- **Tasks:** Миграция, добавление `sent_to_1c`, `sent_to_1c_at`, `status_1c`.
- **Verification:** Поля видны в Django Admin.

#### Story 2: Order Export Service (XML Generation)

- **Goal:** Сервис формирования XML.
- **Tasks:** `OrderExportService`: выборка заказов, генерация XML по схеме. Добавить **валидацию ИНН** (если нет - пробовать отправить без него или с заглушкой/физлицо).
- **Verification:** Unit-тест: создать заказ, вызвать сервис, проверить наличие `<ИНН>` и `<ХозОперация>`.

#### Story 3: Exchange View Logic (mode=query & success)

- **Goal:** Выгрузка по HTTP.
- **Tasks:**
  - `handle_query`: поддержка `zip=yes`, формирование ответа.
  - `handle_success`: проставление `sent_to_1c=True`.
- **Verification:** `curl` запрос возвращает валидный (возможно сжатый) XML.

#### Story 4: Import Status Service (mode=file)

- **Goal:** Обработка обратного потока.
- **Tasks:** `OrderImportService`: парсинг входящего `orders.xml`, поиск заказа по ID, обновление статуса.
- **Verification:** Загрузка XML с новым статусом меняет статус заказа в БД.

#### Story 5: Endpoint справочника статусов (`mode=info`)

- **Goal:** Загрузка справочника статусов сайта в 1С-Битрикс.
- **Tasks:** Добавить `handle_info` в `ICExchangeView`, вернуть XML `<Справочник>` со статусами и пустым блоком `<ПлатежныеСистемы>`.
- **Verification:** `curl` запрос `type=sale&mode=info` возвращает валидный XML, а кнопка "Загрузить с сайта" в 1С заполняет таблицу статусов.

#### Story 6: Integration Testing

- **Goal:** E2E тестирование.
- **Tasks:** Написание теста полного цикла: создание → экспорт → симуляция ответа 1С → импорт.
- **Verification:** CI проходит.

## 7. Risks

| Risk                                        | Mitigation                                                                                                                                                                                                                                  |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| У пользователя нет ИНН                      | В 1С настроить поиск по ФИО, либо передавать "000000000000" для физлиц                                                                                                                                                                      |
| Дублирование заказов                        | 1С сама отслеживает `Ид` и не создает дубли, если GUID совпадает                                                                                                                                                                            |
| Большой размер XML                          | Использовать ZIP-сжатие (FR1.3)                                                                                                                                                                                                             |
| В 1С пришел один документ вместо нескольких | Проверить, что заказ создан после исправления, в корзине были товары разных групп, `OrderItem.vat_rate` заполнен snapshot-ом, а `sub-orders` сгруппированы по `(vat_rate, warehouse_name)`. Старые заказы автоматически не пересчитываются. |
